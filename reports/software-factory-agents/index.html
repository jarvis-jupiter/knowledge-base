AI Software Factory

Designing a multi-agent workflow on OpenClaw — where each agent plays a dedicated role in a fully autonomous software delivery pipeline.

The Concept

A team of agents, each with a job

🧠

The Problem

A single general-purpose agent doing everything — planning, coding, testing — loses specialisation. A human software team works because each role has deep context, tools, and accountability.

🏭

The Insight

OpenClaw agents are fully isolated brains — each gets its own workspace, SOUL.md persona, tool set, and session store. This maps perfectly to a professional role definition.

🔗

The Glue

sessions_send passes structured work between agents. sessions_spawn enables parallelism. A shared filesystem acts as the persistent handoff medium between roles.

📁 Shared Filesystem

All agents read/write to /factory/ — spec files, tickets, review reports. Artefacts persist between turns, no context lost.

⚡ sessions_spawn

Dev can spawn parallel sub-agents for independent modules. Each sub-task runs concurrently and reports back on completion.

🛑 Human gates

Pipeline naturally pauses before any irreversible action — PR merge, deploy, database migration — and waits for Sir's explicit approval.

Handoff Design

The shared filesystem contract

Directory Structure
# /factory/ — shared handoff medium /factory/ ├── tickets/ │ ├── TICKET-001.md ← PM writes │ └── TICKET-002.md ├── specs/ │ └── feature-x.md ← PM writes, Dev reads ├── reviews/ │ └── pr-42-review.md ← QA writes ├── reports/ │ └── sprint-summary.md← PM/QA write └── status.json ← pipeline state
// status.json — pipeline heartbeat { "currentTicket": "TICKET-001", "stage": "qa", "prUrl": "github.com/.../pull/42", "qaVerdict": "pending", "updatedAt": "2026-03-06T16:00Z" }
Why Files Over Messages
📄

Persistence

Files survive agent restarts, compaction, and session gaps. A message in a session can be lost after context window rollover.

🔍

Structured data

Markdown specs and JSON status files are machine-readable. Any agent can parse them without relying on conversational context.

🔗

Audit trail

Every file change is a git commit. The full history of who wrote what, when, is preserved — critical for a production factory.

⚡

Parallel access

Multiple agents can read the same spec simultaneously without waiting. Dev and QA can both inspect the same ticket at once.

Configuration

openclaw.json — wiring it all up

// ~/.openclaw/openclaw.json { agents: { list: [ { id: "main", workspace: "~/.openclaw/workspace" }, { id: "pm", workspace: "~/.openclaw/workspace-pm", tools: { deny: ["exec"] } }, { id: "dev", workspace: "~/.openclaw/workspace-dev", tools: { allow: [ "exec", "read", "write", "edit", "web_search", "sessions_send" ] } }, { id: "qa", workspace: "~/.openclaw/workspace-qa", tools: { allow: ["exec", "read", "sessions_send"], deny: ["write", "edit"] } } ] }, tools: { agentToAgent: { enabled: true, allow: ["main", "pm", "dev", "qa"] } } }
Key Design Points
🔒

Least-privilege tools

PM can't run shell commands. QA can't modify source files. Dev has full exec. Each agent's blast radius is limited to what its role actually requires.

🧠

SOUL.md per agent

Each workspace has a tailored persona file. PM thinks in acceptance criteria; Dev thinks in clean commits; QA thinks in edge cases and coverage.

🔗

agentToAgent enabled

The explicit allowlist is required for sessions_send to cross agent boundaries. Without it, agents can only send messages within their own scope.

📂

Shared /factory dir

All workspaces mount the same /factory/ path. No need to pass full context in messages — agents read the file directly.

Agent Personas

SOUL.md — each agent's identity

🎯

PM Agent SOUL.md

Thinks like a seasoned product manager. Never codes. Never tests. Breaks ambiguity into clarity.

"You are a senior product manager. Your only output is structured specs — epics, user stories, acceptance criteria, and task breakdowns. You never write code. When done, write the spec to /factory/specs/ and notify dev via sessions_send."
💻

Dev Agent SOUL.md

Senior engineer. TDD. Clean commits. Opens PRs. Responds to QA feedback without ego.

"You are a senior software engineer. Read the spec from /factory/specs/. Implement TDD. Run all checks before committing. Open a PR. When done, notify qa via sessions_send with the PR URL. Never merge your own PRs."
🧪

QA Agent SOUL.md

Sceptical. Thorough. Writes structured verdicts. Never lets a flaky pass through.

"You are a senior QA engineer. Review the PR. Run E2E and unit tests. Validate every acceptance criterion in the spec. Write your verdict to /factory/reviews/. On pass: notify main. On fail: notify dev with a bug report."

The SOUL.md is the agent's standing operating procedure. It doesn't need to be told what to do each time — it already knows its role, its outputs, and who to notify when it's done.

Limitations & Roadmap

What works today — what comes next

Current Limitations
⚠

No native pipeline primitive

The PM → Dev → QA chain is orchestrated via sessions_send calls and SOUL.md standing instructions — not a built-in OpenClaw construct. Requires careful persona design to prevent loops.

⚠

No sub-agent spawning by sub-agents

OpenClaw currently prevents sub-agents from spawning their own sub-agents — Dev can't farm out parallel modules unless orchestrated from main.

⚠

Context window budget

Each agent has a finite context window. Long-running tickets require careful file-based handoffs to avoid losing early decisions to compaction.

Recommended Next Steps
1

Bootstrap the three workspaces

Create workspace-pm, workspace-dev, workspace-qa with tailored SOUL.md files and the shared /factory/ directory structure.

2

Wire openclaw.json

Add agents, tool policies, and agentToAgent allowlist. Restart gateway and verify with openclaw agents list --bindings.

3

Pilot on a real ticket

Run a small, well-scoped feature through the full PM → Dev → QA pipeline. Observe where the handoffs break down and iterate on the SOUL.md instructions.

4

Add a /factory command

A single trigger — "build feature X" — that kicks off the entire pipeline from main and delivers a summary to Telegram when QA passes.
